## 1. Принципы модели доступа

### 1.1 Основные принципы

| Принцип | Описание | Применение |
|---------|----------|------------|
| **Принцип минимальных привилегий** | Пользователь получает только те права, которые необходимы для выполнения его обязанностей | Все роли проектируются с минимумом прав |
| **Разделение обязанностей (SoD)** | Критичные операции требуют участия нескольких лиц | Создание и утверждение, создание и удаление |
| **Бизнес-роль первична** | Роль определяется должностными обязанностями, не техническими возможностями | Роли соответствуют бизнес-функциям |
| **Иерархичность** | Высшие роли наследуют права низших | ADMIN → MANAGER → DISPATCHER |
| **Контекстная безопасность** | Права зависят от контекста объекта (свой/чужой) | Ограничение видимости по филиалу |
| **Аудируемость** | Все критичные действия логируются | Журнал аудита для всех операций |

### 1.2 Модель безопасности

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                         МОДЕЛЬ БЕЗОПАСНОСТИ RBAC                                │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│   USER ──── (N:M) ──── USER_ROLE ──── (N:1) ──── ROLE                          │
│   │                                                                           │
│   │                                                                           │
│   └── (N:1) ──── FILIAL (ограничение видимости)                                │
│                                                                                  │
│   ROLE ──── (N:M) ──── ROLE_PERMISSION ──── (N:1) ──── PERMISSION              │
│   │                                                                           │
│   └── (N:M) ──── ROLE_SCOPE (область действия)                                 │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 1.3 Типы доступа к объектам

| Тип доступа | Обозначение | Описание |
|-------------|-------------|----------|
| **Полный доступ** | FULL | Создание, чтение, изменение, удаление |
| **Управление** | MANAGE | Чтение, изменение, управление статусом |
| **Редактирование** | EDIT | Чтение, изменение (без удаления) |
| **Просмотр** | READ | Только чтение |
| **Отсутствие** | NONE | Нет доступа |

### 1.4 Контексты ограничения

| Контекст | Обозначение | Описание |
|----------|-------------|----------|
| **Свой объект** | OWN | Объект, созданный пользователем или назначенный ему |
| **Филиал** | FILIAL | Все объекты филиала пользователя |
| **Регион** | REGION | Все объекты региона пользователя |
| **Система** | SYSTEM | Все объекты без ограничений |

### 2.2 Наследование прав

| Роль | Наследует от | Добавляет к наследованию |
|------|--------------|--------------------------|
| SYSTEM | — | Все права без ограничений |
| ADMIN | SYSTEM | Управление пользователями, настройки |
| MANAGER | DISPATCHER | Отчёты, KPIs, согласования |
| DISPATCHER | OPERATOR | Планирование рейсов, назначение |
| SALES | OPERATOR | Работа с заявками и клиентами |
| OPERATOR | — | Базовые операции ввода |
| DRIVER | — | Только свои рейсы |
| CLIENT_USER | — | Портал клиента (future) |

## 3. Бизнес-роли

### 3.1 Определение бизнес-ролей

| Роль | Код | Описание | Профиль |
|------|-----|----------|---------|
| **Администратор** | ADMIN | Полный доступ к системе | IT/DevOps |
| **Руководитель** | MANAGER | Контроль операций и персонала | Head of operations |
| **Диспетчер** | DISPATCHER | Планирование и управление рейсами | Operations |
| **Менеджер продаж** | SALES | Работа с клиентами и заявками | Sales |
| **Оператор** | OPERATOR | Ввод данных, документация | Back office |
| **Водитель** | DRIVER | Мобильное приложение | Field |
| **Клиент** | CLIENT | Портал для внешних клиентов | External (future) |

### 3.2 Детальное описание ролей

#### ADMIN — Администратор

| Характеристика | Описание |
|----------------|----------|
| **Назначение** | Техническое администрирование системы |
| **Профиль пользователя** | IT-специалист, DevOps-инженер |
| **Опыт работы** | 3+ лет системного администрирования |
| **Типичные действия** | Создание пользователей, настройка интеграций, управление доступом |
| **Ограничения** | Нет операционных ограничений |

**Уникальные права:**
- Управление ролями и разрешениями
- Настройка системных параметров
- Доступ к журналу аудита без ограничений
- Управление интеграциями (API keys)

---

#### MANAGER — Руководитель

| Характеристика | Описание |
|----------------|----------|
| **Назначение** | Стратегическое управление операциями |
| **Профиль пользователя** | Head of operations, Regional manager |
| **Опыт работы** | 5+ лет в логистике |
| **Типичные действия** | Просмотр KPIs, согласование ставок, управление персоналом |
| **Ограничения** | Нет прав на удаление критичных данных |

**Уникальные права:**
- Просмотр всех KPIs
- Согласование ставок
- Управление расписанием подчинённых
- Формирование любых отчётов

---

#### DISPATCHER — Диспетчер

| Характеристика | Описание |
|----------------|----------|
| **Назначение** | Оперативное управление рейсами |
| **Профиль пользователя** | Диспетчер, координатор |
| **Опыт работы** | 1-3 года |
| **Типичные действия** | Планирование рейсов, назначение водителей, отслеживание |
| **Ограничения** | Только свой филиал |

**Уникальные права:**
- Назначение водителей на рейсы
- Назначение транспорта
- Изменение статусов рейсов
- Управление расписанием

---

#### SALES — Менеджер по продажам

| Характеристика | Описание |
|----------------|----------|
| **Назначение** | Работа с клиентами, продажа услуг |
| **Профиль пользователя** | Sales manager, Account manager |
| **Опыт работы** | 1-5 лет |
| **Типичные действия** | Создание заявок, работа с клиентами, формирование коммерческих предложений |
| **Ограничения** | Только свои клиенты |

**Уникальные права:**
- Создание заявок
- Работа с клиентской базой (свои клиенты)
- Формирование предложений
- Клонирование заявок

---

#### OPERATOR — Оператор

| Характеристика | Описание |
|----------------|----------|
| **Назначение** | Ввод данных, подготовка документации |
| **Профиль пользователя** | Оператор, секретарь |
| **Опыт работы** | 0-1 год |
| **Типичные действия** | Ввод данных, оформление документов |
| **Ограничения** | Только чтение без утверждения |

**Уникальные права:**
- Ввод данных в справочники (ограниченный)
- Подготовка документов
- Поиск и просмотр объектов

---

#### DRIVER — Водитель

| Характеристика | Описание |
|----------------|----------|
| **Назначение** | Выполнение рейсов |
| **Профиль пользователя** | Водитель (мобильное приложение) |
| **Опыт работы** | Не требуется |
| **Типичные действия** | Обновление статусов, фиксация точек |
| **Ограничения** | Только свои рейсы |

**Уникальные права:**
- Обновление статуса своих рейсов
- Отметка контрольных точек
- Просмотр маршрута

## 6. Операционные ограничения

### 6.1 Ограничения по объектам (Ownership)

| Тип ограничения | Обозначение | Описание |
|-----------------|-------------|----------|
| **Свои объекты** | OWN | Объект создан пользователем или назначен ему |
| **Филиал** | FILIAL | Все объекты филиала пользователя |
| **Регион** | REGION | Все объекты региона пользователя |
| **Без ограничений** | ALL | Все объекты системы |

### 6.2 Ограничения по действиям

| Действие | Ограничение | Обоснование |
|----------|-------------|-------------|
| Создание заявки | OWN, FILIAL | Контроль нагрузки |
| Изменение заявки | OWN, FILIAL | SoD: нельзя изменять свои же без контроля |
| Удаление заявки | ADMIN, MANAGER | Только руководство |
| Изменение статуса | OWN + MANAGER | SoD: создатель не может сам подтверждать |
| Создание рейса | FILIAL | Только в своём филиале |
| Назначение водителя | DISPATCHER, MANAGER, ADMIN | Контролируемое действие |
| Согласование заказа | MANAGER, ADMIN | Повышенная ответственность |
| Удаление клиента | ADMIN | Защита критичных данных |
| Экспорт данных | MANAGER, ADMIN | Контроль утечки данных |

### 6.3 Правила разделения обязанностей (SoD)

| Конфликтующие операции | Роль A | Роль B | Комментарий |
|------------------------|--------|--------|-------------|
| Создание и утверждение | SALES | MANAGER | Создатель не может сам утвердить |
| Создание и удаление | DISPATCHER | ADMIN | Диспетчер не удаляет |
| Назначение и закрытие | DISPATCHER | DISPATCHER | Разные диспетчеры |
| Выставление и оплата | SALES | FINANCE | Разные отделы |

## 9. Ограничения по филиалам и регионам

### 9.1 Модель ограничений

```
┌─────────────────────────────────────────────────────────────────────────────────┐
│                     МОДЕЛЬ ОГРАНИЧЕНИЙ ПО ФИЛИАЛАМ                             │
├─────────────────────────────────────────────────────────────────────────────────┤
│                                                                                  │
│   USER (Москва) ──── scope: FILIAL ──── USER ↔ FILIAL                        │
│   │                                                                           │
│   └── Видит только объекты FILIAL = USER.filial_id                            │
│                                                                                  │
│   USER (Head) ──── scope: REGION ──── USER ↔ REGION                          │
│   │                                                                           │
│   └── Видит объекты всех FILIAL WHERE region_id = USER.region_id              │
│                                                                                  │
│   USER (Admin) ──── scope: SYSTEM ──── Без ограничений                        │
│   │                                                                           │
│   └── Видит все объекты                                                        │
│                                                                                  │
└─────────────────────────────────────────────────────────────────────────────────┘
```

### 9.2 Типы областей действия (Scope)

| Scope | Описание | Применимые роли |
|-------|----------|-----------------|
| SYSTEM | Без ограничений | ADMIN, SYSTEM |
| REGION | Все филиалы региона | MANAGER (regional) |
| FILIAL | Только свой филиал | DISPATCHER, SALES, OPERATOR |
| OWN | Только свои объекты | OPERATOR, DRIVER |

### 9.3 Ограничения по типу операции

| Операция | Scope | Описание |
|----------|-------|----------|
| Создание заявки | FILIAL | Заявка создаётся в филиале пользователя |
| Создание рейса | FILIAL | Рейс создаётся в филиале пользователя |
| Назначение водителя | FILIAL | Водитель должен быть из того же филиала |
| Назначение транспорта | FILIAL | Транспорт должен быть из того же филиала |
| Назначение ответственного | REGION | Ответственный может быть из региона |
| Просмотр заявок | REGION | MANAGER видит все филиалы региона |
| Формирование отчётов | REGION | MANAGER видит данные региона |

### 9.4 Исключения из ограничений

| Ситуация | Исключение | Кто может |
|----------|------------|-----------|
| Межфилиальная перевозка | Рейс может захватывать объекты нескольких филиалов | DISPATCHER+ |
| Передача заявки | Заявка может быть передана другому филиалу | MANAGER+ |
| Назначение ответственного | Ответственный может быть из любого филиала | MANAGER+ |
| Аварийная ситуация | ADMIN может работать с любым филиалом | ADMIN |

## 10. Спорные зоны и вопросы

### 10.1 Вопросы, требующие уточнения

#### Вопрос 1: Права SALES на изменение заявок

**Проблема:**
- SALES создаёт заявку → становится её владельцем
- SALES может изменять свои заявки без ограничений
- Но: может ли SALES менять статус заявки?

**Варианты решения:**

| Вариант | Описание | Риски |
|---------|---------|-------|
| A | SALES может менять статус до "Подтверждена" | Может создать конфликт с диспетчером |
| B | SALES не может менять статус (только MANAGER/DISPATCHER) | Задержка в обработке |
| C | SALES может менять только "свои" поля (контактные данные) | Сложная реализация |

**Рекомендация:** Вариант B (SALES не меняет статус)

---

#### Вопрос 2: Самостоятельная отмена заявки

**Проблема:**
- Может ли DISPATCHER самостоятельно отменить заявку?
- Или требуется согласование MANAGER?

**Варианты решения:**

| Вариант | Описание | Обоснование |
|---------|---------|-------------|
| A | DISPATCHER может отменять сам (с причиной) | Оперативность |
| B | DISPATCHER + согласование MANAGER | Контроль |
| C | MANAGER только | Избыточный контроль |

**Рекомендация:** Вариант B — отмена требует согласования или причины

---

#### Вопрос 3: Права OPERATOR на ввод данных

**Проблема:**
- OPERATOR должен вводить данные в справочники?
- Какие справочники?

**Варианты решения:**

| Справочник | OPERATOR может | Обоснование |
|------------|----------------|-------------|
| Клиенты | Read only | Критичные данные |
| Точки | Create, Update | Операционные данные |
| Типы грузов | Read only | Конфигурационные |
| Водители | Read only | HR данные |
| Транспорт | Read only | Fleet данные |

**Рекомендация:** OPERATOR → ограниченный ввод только в точки и комментарии

---

#### Вопрос 4: Диспетчер и чужие рейсы

**Проблема:**
- DISPATCHER_A создал рейс
- DISPATCHER_B может его редактировать?

**Варианты решения:**

| Вариант | Описание | Проблема |
|---------|---------|----------|
| A | Все DISPATCHER одного филиала равны | Отсутствие ответственности |
| B | Только создатель и MANAGER | Негибкость при болезни |
| C | Создатель или MANAGER | Лучший баланс |

**Рекомендация:** Вариант C + логирование всех изменений

---

#### Вопрос 5: Водитель и изменение статуса рейса

**Проблема:**
- DRIVER может подтвердить выезд (OK)
- DRIVER может отметить точку (OK)

**Варианты решения:**

| Вариант | Описание | Риски |
|---------|---------|-------|

## Приложения

### A. Глоссарий RBAC

| Термин | Определение |
|--------|-------------|
| **RBAC** | Role-Based Access Control — управление доступом на основе ролей |
| **Permission** | Разрешение на выполнение конкретного действия |
| **Role** | Набор разрешений, объединённых общей бизнес-функцией |
| **Scope** | Область действия разрешения (система, регион, филиал, свой) |
| **SoD** | Separation of Duties — разделение обязанностей |
| **Ownership** | Принадлежность объекта пользователю |

### B. Рекомендации по имплементации

1. **Роли** — предопределённые, хранятся в базе данных
2. **Разрешения** — предопределённые enum-значения
3. **Связь** — USER_ROLE промежуточная таблица (N:M)
4. **Кэширование** — права кэшируются в сессии (TTL: 15 мин)
5. **Аудит** — все изменения прав логируются

### C. Тестирование RBAC

| Тип теста | Описание | Метод |
|-----------|----------|-------|
| Unit | Каждая роль имеет ожидаемые права | Автоматический |
| Integration | Комбинации ролей и действий | Таблица решений |
| Security | Попытки несанкционированного доступа | Penetration testing |
| SoD | Проверка разделения обязанностей | Автоматический |
