1. Цель продукта
Заявление о продукте
Единая платформа для управления логистическими операциями, обеспечивающая:
- Централизацию — все заявки, заказы и рейсы в одной системе
- Прозрачность —实时ный контроль статусов и местоположения грузов
- Автоматизацию — сокращение рутинных операций диспетчеров
- Аналитику — контроль эффективности и принятие data-driven решений
Целевые метрики (KPI продукта)
Метрика
Время обработки заявки
Ошибки в оформлении
Процент выполненных в срок
Время формирования отчётов

2. Бизнес-проблемы
2.1 Проблемы операционной деятельности
#	Проблема
P1	Фрагментация данных — информация о заявках в Excel, email, звонках
P2	Отсутствие единого статуса — клиент не понимает, где его груз
P3	Ручное распределение ресурсов — назначение водителей и транспорта "по памяти"
P4	Нет контроля сроков — дедлайны отслеживаются в таблицах
P5	Разрозненные каналы коммуникации — email, телефон, мессенджеры
2.2 Проблемы управления
#	Проблема
P6	Отсутствие аналитики — решения принимаются на основе "чутья"
P7	Нет единого окна для руководителя — KPI считаются вручную
P8	Сложность масштабирования — при росте бизнеса растёт хаос
2.3 Боли пользователей (Jobs to Be Done)
Когда я получаю новую заявку от клиента,
я хочу быстро её оформить и найти подходящий транспорт,
чтобы не потерять клиента и вписаться в сроки.
Когда мой клиент спрашивает статус,
я хочу ответить за 30 секунд,
чтобы не тратить время на поиск информации.
Когда начальство спрашивает эффективность за месяц,
я хочу нажать одну кнопку,
чтобы не сидеть 3 дня с Excel.

3. Основные пользовательские сценарии
UC-1: Обработка входящей заявки
Актор: Менеджер по работе с клиентами
Триггер: Звонок / email / форма на сайте
Предусловие: Клиент обратился с потребностью в перевозке
Основной поток:
1. Менеджер создаёт заявку в CRM
2. Заполняет параметры груза (вес, объём, тип)
3. Указывает точки загрузки и выгрузки
4. Система предлагает расчёт (авто/жд)
5. Менеджер согласовывает с клиентом
6. Заявка переходит в статус "Подтверждена"
Альтернативные потоки:
- 4a. Расчёт невозможен → эскалация старшему менеджеру
- 5a. Клиент отказывается → заявка архивируется
Постconditions:
- Заявка создана со статусом и ответственным
- Клиенту отправлено подтверждение
UC-2: Планирование рейса
Актор: Диспетчер
Триггер: Поступление подтверждённой заявки
Предусловие: Заявка имеет статус "Подтверждена"
Основной поток:
1. Диспетчер видит заявку в очереди планирования
2. Открывает карточку, видит требования к транспорту
3. Подбирает доступный транспорт и водителя
4. Назначает время погрузки с учётом расписания
5. Создаёт рейс, связывает с заявкой
6. Отправляет задание водителю (push/email)
Результат:
- Рейс создан со статусом "Запланирован"
- Водитель уведомлён
- Статус заявки обновлён
UC-3: Отслеживание в пути
Актор: Диспетчер / Менеджер
Триггер: Рейс начат
Предусловие: Водитель подтвердил выезд
Основной поток:
1. Система фиксирует выезд со склада (ручной ввод или интеграция)
2. Водитель обновляет статусы по этапам:
   - Погрузка завершена
   - В пути
   - Прибытие на точку
   - Выгрузка завершена
3. Каждый статус фиксирует время и геолокацию
4. Система автоматически уведомляет клиента при смене статуса
Результат:
- Полная история движения груза
- Клиент имеет актуальную информацию
UC-4: Закрытие и аналитика
Актор: Диспетчер / Руководитель
Триггер: Рейс завершён
Предусловие: Выгрузка подтверждена
Основной поток:
1. Водитель фиксирует завершение (приложение/звонок)
2. Диспетчер проверяет данные, закрывает рейс
3. Система рассчитывает фактические показатели:
   - Время в пути vs план
   - Расход топлива (если доступно)
   - Отклонения от маршрута
4. Рейс попадает в аналитику
Результат:
- Рейс закрыт
- Данные доступны для отчётности

4.3 Пользовательские истории ролей
Диспетчер:
> Как диспетчер, я хочу видеть все активные рейсы на одной карте, чтобы быстро выявлять проблемные точки и принимать решения о перераспределении.
Менеджер по продажам:
> Как менеджер, я хотеть видеть историю всех коммуникаций с клиентом в одной карточке, чтобы не переключаться между email и телефоном.
Руководитель:
> Как руководитель, я хочу видеть дашборд с ключевыми KPIs в реальном времени, чтобы контролировать бизнес без запросов к подчинённым.

5.2 Детальное описание модулей
Модуль 1: Dashboard (Рабочий стол)
Назначение: Единая точка входа, показывающая актуальную информацию для каждой роли.
Функции:
- Персонализированные виджеты по ролям
- Очередь задач (заявки на обработку)
- Активные рейсы сегодня
- Просроченные задачи
- Быстрые действия (создать заявку, найти водителя)
- Уведомления (новые заявки, смены статусов)
- Погода и пробки (влияние на маршруты)
Виджеты:
Виджет
Мои задачи на сегодня
Активные рейсы
Новые заявки (24ч)
KPIs дня
Незавершённые рейсы
Карта с рейсами
---
Модуль 2: Orders (Заказы)
Назначение: Управление коммерческими заказами — от создания до исполнения.
Функции:
- Создание заказа (ручное + импорт)
- Привязка к клиенту и договору
- Расчёт стоимости (тарифы, наценки, скидки)
- Формирование документации (счёт, акт)
- Отслеживание оплаты
- Связь с заявкой на перевозку
Статусы заказа:
Черновик → Согласован → Оплачен → Частично исполнен → Исполнен → Архив
---
Модуль 3: Requests (Заявки)
Назначение: Ядро системы — управление заявками на перевозку.
Функции:
- Создание заявки (форма, import, API)
- Параметры груза (вес, объём, тип, упаковка, опасность)
- Точки маршрута (загрузка, промежуточные, выгрузка)
- Требования к транспорту (тип, вместимость, температура)
- Выбор типа перевозки (авто, жд, мультимодал)
- Расчёт времени и стотусов
- Флаги и приоритеты
- История изменений
Статусы заявки:
Новая → Расчёт → Предложена → Подтверждена → В работе → Завершена → Отменена
Флаги:
Флаг	Обозначение
Срочная	🔴 URGENT
Негабарит	🟠 OVERSIZE
Хрупкий	🔵 FRAGILE
Температурный	❄️ TEMP
Опасный	☣️ HAZMAT
Экспресс	⚡ EXPRESS
---
Модуль 4: Trips (Рейсы)
Назначение: Планирование и отслеживание рейсов.
Функции:
- Создание рейса из заявки
- Назначение транспорта и водителя
- Планирование маршрута
- Отслеживание в реальном времени
- Фиксация контрольных точек
- Обновление ETA
- Закрытие рейса с отчётностью
Статусы рейса:
Запланирован → В пути → На погрузке → В пути → На выгрузке → Завершён
                              ↓
                         Проблема (задержка, ДТП, поломка)
---
#### Модуль 5: Schedule (Расписание)
**Назначение:** Управление расписанием и ресурсами.
**Функции:**
- Календарь рейсов
- График работы водителей
- Доступность транспорта
- Планирование ТО
- Учёт выходных и праздников
**Виды отображения:**
- Календарь (день, неделя, месяц)
- Gantt-диаграмма
- Список с фильтрами
---
Модуль 6: Railway (ЖД перевозки)
Назначение: Управление железнодорожными перевозками.
Функции:
- Интеграция с ЖД системами (вагоны, контейнеры)
- Отслеживание поездов
- Управление контейнерными площадками
- Координация с РЖД/операторами
Особенности:
- Длинные сроки доставки (дни vs часы)
- Другие статусы и точки контроля
- Специфические справочники

Функции:

Модуль 11: KPI (Ключевые показатели)
Назначение: Мониторинг эффективности.
KPI Dashboard:
KPI	Формула
Выполнение в срок	Рейсы в срок / Все рейсы
Загрузка флота	Факт.тонны / Макс.тонны

Процент срывов	Отмены / Все заявки
Конверсия в рейс	Рейсы / Заявки
Визуализации:
- Тренды (линейные графики)
- Распределение (pie, bar)
- Сравнение периодов
- Heatmaps (по филиалам, водителям)

7. MVP Scope
7.1 Определение MVP
MVP (Minimum Viable Product) — первая работоспособная версия системы, решающая основные проблемы клиента и позволяющая валидировать продукт на реальных пользователях.
7.2 Входящие в MVP модули
Модуль	Приоритет
Dashboard	P0
Requests (заявки)	P0
Trips (рейсы)	P0
Reference: Clients	P0
Reference: Vehicles	P0
Reference: Drivers	P0
Notifications	P1
Auth & Users	P0
Admin (базовый)	P0
7.3 Функциональность MVP
Dashboard
- [ ] Виджет "Мои задачи" для текущего пользователя
- [ ] Виджет "Активные рейсы сегодня"
- [ ] Виджет "Просроченные задачи"
- [ ] Базовые уведомления
- [ ] Быстрые действия (создать заявку)

9. Риски, допущения и вопросы (RAID)
9.1 Риски (Risks)
ID	Риск
R1	Недостаточное вовлечение пользователей — ключевые пользователи не будут использовать систему
R2	Сложность миграции данных — данные в Excel не перенесутся чисто
R3	Неопределённость бизнес-процессов — процессы изменятся в ходе разработки
R4	Масштабируемость — система не выдержит роста
R5	Безопасность — утечка данных клиентов
R6	Integration delays — внешние интеграции задержат релиз
R7	Performance — медленная работа на данных > 10K записей
9.2 Допущения (Assumptions)
ID	Допущение
A1	CRM будет использоваться 50+ пользователями ежедневно
A2	Текущие бизнес-процессы можно формализовать
A3	Клиенты согласны на email + web уведомления
A4	Водители будут пользоваться базовым функционалом обновления статусов
A5	IT-инфраструктура клиента поддерживает современный браузер
A6	MVP 3-4 месяца — достаточный срок
A7	Доступен эксперт по логистике для консультаций

C. Definition of Done (критерии завершённости)
Для включения в релиз функция должна:
1. Иметь работающий UI согласно дизайну
2. Сохранять данные в базу
3. Иметь валидацию входных данных
4. Обрабатывать ошибки (graceful degradation)
5. Пройти code review
6. Пройти функциональное тестирование
7. Иметь базовую документацию
8. Соответствовать accessibility (WCAG 2.1 AA)
